# AI Agent 应用层学习总结：Harness / Loop / Headless

> 学习时间：2026.06.24
> 学习工具：Claude Code CLI + 手写 Demo 实验

---

## 一、三个概念的关系

```
┌─────────────────────────────────────────────────────┐
│  Headless（部署层）                                   │
│  无界面运行，事件/定时/API 触发                        │
│                                                      │
│  ┌─────────────────────────────────────────────┐     │
│  │  Harness（管控层）                            │     │
│  │  工具注册、权限控制、循环限制、日志审批         │     │
│  │                                              │     │
│  │  ┌─────────────────────────────────────┐     │     │
│  │  │  Loop（执行层）                      │     │     │
│  │  │  思考 → 行动 → 观察 → 循环           │     │     │
│  │  └─────────────────────────────────────┘     │     │
│  └─────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────┘
```

**一句话总结：Loop 决定"怎么干活"，Harness 决定"能干什么"，Headless 决定"怎么启动"。**

---

## 二、Loop — Agent 的核心执行引擎

### 定义
Agent 反复执行"思考 → 行动 → 观察"循环，直到任务完成或达到停止条件。

### 实际观察（Claude Code CLI）

给 Claude Code 输入任务后，观察到的执行流程：

| 阶段 | Claude Code 显示 | 含义 |
|------|-----------------|------|
| 思考 | Thought 3s | LLM 推理，决定下一步 |
| 准备 | Brewed 9s | 构建执行计划 |
| 行动 | Bash(find...) | 调用工具执行命令 |
| 观察 | Churned 16s | 分析执行结果 |
| 输出 | Recap | 返回最终结论 |

### 关键特征
- 每轮 Loop = 一次 LLM 调用 + 一次工具执行
- Loop 可以嵌套多轮（第一轮发现路径不对 → 第二轮换路径继续）
- 停止条件：任务完成 / 最大轮次 / 遇到不可恢复的错误

## 三、Harness — Agent 的安全管控框架

### 定义
包裹 Loop 的确定性程序逻辑，负责管控 Agent 的行为边界。

### 四个核心能力

| 能力 | 作用 | 实验验证 |
|------|------|---------|
| 工具注册表（Tool Registry） | 定义 Agent 能用哪些工具 | 定义了 queryJira/analyzeCause/pushWecom |
| 最大循环次数 | 防止无限循环 | maxIterations 从 5 改为 3，第 4 轮被强制停止 |
| Lifecycle Hooks | 执行前后的日志/审批 | beforeAction 检查权限，pushWecom 触发审批提示 |
| 错误处理和重试 | 工具失败时自动重试 | retryLimit: 2，失败重试两次后放弃 |

### 实验结果

**实验 1：maxIterations = 3**
```
第 1 轮 ← queryJira 成功
第 2 轮 ← analyzeCause 成功  
第 3 轮 ← analyzeCause 成功
⛔ [Guardrail] 达到最大循环次数 3，强制停止！
→ pushWecom 没有执行
```

**实验 2：allowedTools 中删除 pushWecom**
```
第 4 轮 → 准备调用 pushWecom
❌ [Guardrail] 工具 "pushWecom" 不在允许列表中，已拦截！
→ 工具函数根本没被调用
```

### Claude Code 中的 Harness 体现
- `/permissions` 命令 → 查看工具注册表和权限规则
- 删除文件时弹出确认 → beforeAction Hook 的审批机制
- Allow / Ask / Deny 三个权限级别 → Guardrail 分层管控

## 四、Headless — Agent 的无界面部署模式

### 定义
Agent 在后台自主运行，不需要用户界面交互，触发条件满足时自动启动。

### 三种触发方式

| 触发方式 | 机制 | 应用场景 |
|---------|------|---------|
| ⏰ 定时 Cron | 到点自动执行 | 每天9点停滞预警推送 |
| 📨 事件 Event | Webhook/消息队列 | Jira 状态变化时实时响应 |
| 📡 API 调用 | REST 接口 | 其他系统按需调用分析能力 |

### 与传统模式的对比

| | 传统模式 | Headless Agent |
|--|---------|----------------|
| 触发 | 人输入问题 | 定时/事件/API 自动触发 |
| 执行 | 人等待回复 | Agent 自主完成多步任务 |
| 输出 | 显示在界面上 | 推送通知/写入系统/返回 API |
| 人的参与 | 全程在线 | 只在异常时介入 |

## 五、与 AI-PM 平台的结合分析

### 现状

| 能力 | 状态 | 说明 |
|------|------|------|
| Headless 定时触发 | ✅ 已有 | server.js 每天9点推送停滞 tickets |
| Loop 多轮执行 | ❌ 缺失 | 只是查数据+推送，没有 LLM 推理分析 |
| Harness 管控 | ❌ 缺失 | 没有工具注册、权限控制、循环限制 |
| 事件触发 | ❌ 缺失 | 没有接 Jira Webhook 实时响应 |
| API 触发 | ❌ 缺失 | 没有暴露 Agent 分析端点 |

### 升级路径

```
Phase 1: 给定时推送加上 LLM 分析能力
  现在: 查停滞 tickets → 直接推送列表
  目标: 查停滞 tickets → LLM 分析原因 → 生成建议 → 推送智能报告

Phase 2: 搭建 Harness 管控框架
  - 抽象工具层（Jira/企微/LLM 统一注册）
  - 加入循环限制、权限控制、执行日志
  - 关键操作（修改 Jira、推送通知）需要管控

Phase 3: 扩展 Headless 触发方式
  - 接入 Jira Webhook → 实时风险响应
  - 暴露 /api/agent/analyze 端点 → 其他系统可调用
  - 企微消息触发 → 群内 @ Bot 即可发起分析
```

## 六、核心认知

1. **Loop 是 Agent 的大脑** — 没有 Loop 就是死脚本，有了 Loop 才能自主推理
2. **Harness 是 Agent 的安全带** — 没有 Harness 的 Agent 像没有刹车的车
3. **Headless 是 Agent 的工作模式** — 让 AI 像 7×24 值班员工一样自动运转
4. **三者不是独立技术，是组合架构** — Loop 在最内层执行，Harness 包裹管控，Headless 决定部署方式
5. **AI-PM 平台已有 Headless 雏形** — 加上 Loop（LLM）和 Harness（管控）就是完整的 Agentic 系统
