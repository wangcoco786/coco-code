# AI-PM & FDE 转型方案

> **FDE = Forward Deployed Engineer**（前线部署工程师）
> 核心理念：用 AI 把"管理"交给平台，把人的精力集中在"交付"上——交付客户价值、交付 AI 转型成果。

---

## 一、AI-PM 转型总览

### 1.1 转型定位

从"对内管理 24 个部门"转变为"对外直接服务客户"。转型核心不是"从 PM 变成更好的 PM"，而是**从对内管理者变为对外交付者**。

| 维度 | 现状（对内 PM） | 转型后（对外 FDE） |
|------|-------------|--------------|
| 服务对象 | 内部 24 个开发部门 | 外部客户 |
| 核心产出 | 内部报告、协调纪要 | 客户方案、可运行 Demo、系统交付 |
| 管理方式 | 人工跟进 Jira | AI-PM 平台自动化 |
| 技术深度 | 仅需求层面 | 能独立实现中等需求、主导系统联调 |
| 价值定位 | 成本中心（管理开销） | 利润中心（直接创收） |

### 1.2 三条主线

1. **AI 驱动对内管理**：AI-PM 平台 → 各部门自组织自管理 → 绩效透明化 → 无需人工 PM 介入
2. **AI 赋能对外交付**：AI 辅助需求分析 → AI 快速出方案/Demo → 分级交付 → 直接创造商业价值
3. **AI 转型推进**：自己先用 → 验证效果 → 组织每周二 AI 会议 → 输出报告 → 推动全公司提效

### 1.3 AI-PM 平台如何实现自组织自管理

核心理念：**开发团队不再需要 PM 来"管"，通过 AI-PM 平台实现自组织、自管理。**

| 平台能力 | 如何支持自管理 | 替代的 PM 工作 |
|---------|------------|-----------|
| Sprint 看板 | 5 列看板（待办→进行中→评审→测试→完成），状态一目了然 | PM 每天跟进进度 |
| 资源视图 | 团队负责人直接看到谁过载/谁空闲，自行调整 | PM 协调资源分配 |
| AI 变更检测 | 自动识别需求变更、范围蔓延(>20%警告)，团队自行响应 | PM 人工发现变更 |
| 绩效排行榜 | SPACE+DORA 五维度评估，24 部门实时排名 | PM 做绩效统计 |
| 风险自动推送 | 延期/阻塞/超负荷自动推送企微群，团队自行解决 | PM 识别和跟进风险 |
| 报告自动生成 | 日报/周报/Sprint 复盘一键生成 | PM 写报告 |
| AI 规划推荐 | AI 从 Backlog 评分推荐候选任务，团队自行选择 | PM 安排任务 |

**管理层监控（无需 PM 中转）：**
- Dashboard 全局视图 → 管理层一屏看全局
- 部门绩效排行榜 → 直观了解各团队表现
- 异常自动预警 → 推送到管理层企微群

> **结论：AI-PM 平台让对内管理从"PM 推动"变为"平台驱动、团队自驱"，PM 角色不再是必须。**

### 1.4 快速交付能力：AI + PM = FDE

#### 分级交付机制

| 需求复杂度 | 判断标准 | 处理方式 | 交付周期 |

- **数据可视化**：ECharts，擅长仪表盘和数据看板
- **Ontology 本体知识库**：AI 直接读取业务字典，自动理解业务模型，快速生成贴合业务的 Demo

#### AI 提升交付效率

| 交付环节 | AI 赋能后 | 效率提升 |
|---------|---------|---------|

> **核心优势：通过 Ontology + AI，可以快速理解任何业务线的数据模型和流程，直接生成与客户业务贴合的可运行 Demo。AI-PM 平台本身就是用 AI 工具开发的，证明了 PM + AI 可以独立完成中等复杂度的产品交付。**

## 二、Glory 客户 — 全球供应链与系统重构（Henan Factory）

> AI-PM 转型视角 · 对外交付型 PM 如何赋能跨国供应链项目

### 2.1 业务背景

河南工厂作为全球皮料供应商，涉及原材料粗加工与成品生产。材料采用双单位管理（平方英尺/公斤 + 序列号追踪），成品带有全球唯一 UPC 码。

**核心诉求**：支持复杂的跨国供应链路由（直发、规避关税的中转、海外前置仓），实现 AX/EDI/OMS/WMS 核心链路重构。

### 2.2 核心履约场景（OMS 自动化分配）

| # | 路由场景 | 说明 |
|---|--------|------|
| 1 | 直发客户 (Direct) | 如 Costco 订单直发 |
| 2 | 关税规避中转 | 成品运至泰国深加工，以"Made in Thailand"名义出口美国 |
| 3 | 跨国前置仓 | 发往欧洲、日本等地备货 |
| 4 | 纯中转 | 中转工厂处理流程 |

**异构系统协同**：
- 美国仓使用标准 WMS
- 泰国/河南工厂使用 Vendor Portal（简化进销存）

**双单位/SN 管理**：材料出库需换算（如 18.5 尺需匹配多张皮料 SN），成品走 UPC 管理

### 2.3 核心系统链路（AX ↔ EDI ↔ OMS ↔ WMS）

```
AX (ERP) → EDI (数据交换枢纽) → OMS (订单管理中枢) → WMS/Vendor Portal (仓库/工厂)
         ←  856(ASN)回传     ←  DC/RC 回传         ←  出入库确认
```

**EDI 数据交换枢纽**：
- AX 原始订单(850 PO) → 标准 JSON 转换下发 OMS
- EDI 856(ASN) 发货通知报文回传 AX
- 多报文格式适配（XML/JSON/Flat File）

**OMS 订单管理中枢**：
- 下发侧：SKU 映射 + 路由分仓（四种场景）+ 幂等防重
- 回传侧：双单位/短装校验 + 异常重试 + 拦截判定
- 订单全生命周期状态管理
- 多仓多路由自动化分配规则引擎

### 2.4 AI-PM 赋能方式

| 职责 | 具体内容 |
|------|---------|
| 🤖 跨国业务需求澄清 | 直接参与客户会议，拉通美国运营、河南/泰国工厂，AI + Ontology 快速理解供应链数据模型 |
| 🤖 异构系统联调统筹 | AI-PM 平台自动跟踪 AX/EDI/OMS/WMS 数据流转测试进度，异常自动推送 |
| 🤖 异常场景闭环 | AI 辅助识别内调单拦截、Partial Shipment、多 SN 匹配等边缘场景，自动生成测试用例 |
| 🆕 快速 Demo 输出 | 1-3 天内输出 OMS 路由配置可视化 Demo，客户可见效果 |
| 🤖 进度与风险把控 | AI-PM 平台自动监控核心节点，从"事后救火"转为"事前预警" |

## 四、LinkW 业务 — YMS 园区管理

> 跨境物流 3PL 平台 · 仓库自动化 · 园区智能调度

### 4.1 业务背景

**小品牌跨境物流痛点**：无法整箱发货，只能依靠托盘或小件零散运输，物流成本高且效率低。

**LinkW 平台解决方案**：
- 订单整合：整合国内小商户订单，集中订舱发货至美国仓库
- 运输与仓储：下游对接 Amazon、Costco 等零售商
- Dropship（一件代发）：减少店家库存压力，避免库存积压

**全局路由**：
```
LinkW (3PL) 国内订单集拼 → 美国转运中心 → Amazon (FBA) / Costco (Dropship)
```

**仓库运转**：
```
集装箱/Trailers(等待区) → 调度 → 月台 DOCK → 自动化分拣与存取区 → 出库发货
```

### 4.2 YMS vs WMS 系统功能对比

| 功能模块 | WMS 现有能力 | YMS 核心价值 |
|---------|-----------|-----------|
| 预约管理 (Appt) | ✅ 支持入库/出库预约，关联PO/SO和承运商 | ❌ 不包含业务单据层面的时段规划 |
| 月台管理 (Dock) | ⚠️ 部分支持（基础指派，缺乏动态排队逻辑） | ✅ 核心能力：基于任务动态指派、冲突检测与实时调度 |
| 入场签到 (Check-In) | 人工操作，登记基础信息 | ✅ 自动化登记（车牌、封签、温度记录等），分配排队号 |
| 场区调度 (Shuttle) | ❌ 不支持 | ✅ 管理穿梭车任务（等待区→月台），监控车辆滞留 |

### 4.3 YMS 业务流程（5 步闭环）

1. **Check-in**：司机到达道闸，YMS 完成登记，记录车辆状态并生成排队号
2. **月台分配**：YMS 根据 Dock 占用情况，下发驶入指令或分配至 Buffer 停车区
3. **Shuttle 调度**：月台释放后，YMS 触发 Shuttle 任务，引导车辆靠泊对应月台
4. **WMS 接管**：WMS 接管实物作业（装卸货），完成后通知 YMS
5. **Check-out**：车辆驶离月台，YMS 完成闭环记录

### 4.4 AI-PM 参与方式

- 负责实地勘察仓库月台物理环境，将现实动线与系统模型映射
- 主导 WMS（作业）与 YMS（调度）交接节点测试
- AI + Kiro 快速搭建月台调度可视化 Demo

> 🎯 **FDE 终极目标**：直接对接仓库现场团队，独立完成月台调度方案落地、设备部署验收、系统上线交付。

## 五、YMS-Box 上线实施 — 司机自动化 Check-in

> 核心目标：实现仓库司机自动化 Check-in，减少人力 · 硬件外采 + 软件自研

### 5.1 技术方案

通过部署边缘工控机（YMS-Box）+ 摄像头，实现物流园区司机自动化 Check-in，取代人工登记流程。

**通信架构**：采用 MQTT 协议（EMQX Broker）实现指令交互 + TCP/IP 双向通信实现视频流传输，解决原有 HTTP 方案的域名爆炸问题。

```
Web 浏览器 / 手机 APP
        ↕ mqtt/tcp ip/http
    YMS Server（消息中台 + 视频中台 + 远程桌面服务）
        ↕ MQTT / TCP/IP / Rustdesk
    YMS-Box 工控机（边缘设备）
        显示程序(H5) | Msg Client | Video Client | Cmd Control | Local Service | Rustdesk
```

### 5.2 技术要点

- **MQTT 指令交互**：YMS 服务端通过 EMQX Broker 与边缘工控机双向通信，下发开启视频流等指令
- **设备状态监控**：EMQX 实时获取工控机在线/离线状态，离线自动告警
- **TCP/IP 视频流**：摄像头实时视频采用 TCP/IP 双向通信，无需申请域名和固定 IP
- **通信安全**：设备握手携带软串号+硬串号验证 → Topic 级别消息隔离与授权

### 5.3 AI-PM 全流程职责

| 实施阶段 | AI-PM 职责 | AI 赋能方式 |
|---------|----------|----------|
| 方案评审 | 参与 MQTT/TCP/IP 架构方案评审，确认通信安全方案 | AI 辅助分析方案文档，快速理解 IoT 架构 |
| 硬件采购协调 | 对接工控机/摄像头厂家，确认设备规格与硬串号管理流程 | AI-PM 平台跟踪采购节点 |
| 现场勘察 | 实地勘察各仓库道闸位置、网络环境、电力条件 | AI 辅助生成点位规划文档与设备清单 |
| 设备激活测试 | 主导工控激活签发流程测试 | AI 辅助生成激活测试用例矩阵 |
| 通信联调 | 验证 MQTT 连接稳定性、视频流传输质量 | AI-PM 平台自动跟踪联调进度，异常自动推送 |
| 上线部署 | 协调各仓库分批部署计划；现场验收 | AI 辅助生成部署 Checklist 与验收标准 |
| 上线后跟踪 | 监控设备在线率，跟踪 Check-in 成功率 | AI 自动计算成功率 & 人力节省数据 |

### 5.4 预期收益

- **人力节省**：取消道闸人工登记岗位，每个仓库减少 1-2 名现场人员
- **效率提升**：司机 Check-in 从人工 3-5 分钟 → 自动化 30 秒内
- **运维成本降低**：无需为每台设备申请域名/固定 IP
- **远程监控**：支持远程查看设备状态和实时视频

### 5.5 风险关注

- 网络稳定性：仓库现场网络环境复杂，需提前验证 MQTT/TCP 连通性
- 设备兼容性：不同厂家工控机需确保 YMS-Box 软件兼容
- 离线处理：工控机断网/断电后的自动恢复与数据补传机制
- 多站点部署：需制定分批上线计划，避免同时部署导致支持资源不足

> 🎯 **FDE 终极目标**：直接对接各站点仓库经理，独立完成 IoT 设备选型→部署→调试→验收全流程交付，每个仓库减少 1-2 名人力。
